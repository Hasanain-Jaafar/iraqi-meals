# CLAUDE.md — مطبخنا (Matbakhna)

This file is the operating manual for any Claude instance (or other agent)
working in this repo. It is organized into pillars, each one a self-contained
unit of context. Read the pillars relevant to your current task before
writing code — most bugs in projects like this come from skipping the
"why," not the "what."

**Project in one sentence:** a static, mobile-first, Arabic-RTL recipe site
for Iraqi home cooking, built on Next.js 16 + Tailwind v4 + shadcn-style
components, deployed as a fully static export to Netlify, with no backend —
the "database" is a JSON file shipped at build time.

---

## Pillar 0 — Source of truth documents

Three files in the repo root are the actual spec. Don't improvise visual
design or data shape from memory — check these first:

| File | What it governs |
|---|---|
| `design.md` | Every color, radius, spacing, glass effect, button rule, and "hard rule" lesson. This is the design system. Treat hex values and class recipes in it as copy-paste source, not inspiration. |
| `design_template_.png` | A directional mockup of 3 screens (home, browse-all, meal detail) plus a "card layout patterns" section. **This is a vibe reference, not a literal spec** — e.g. it shows 4 category filter tabs, but the real data has 10 categories (see Pillar 3). When the mockup and the real data conflict, the data wins; the mockup only governs layout/spacing/visual language. |

Genuinely worth a glance if something behaves differently than expected — Next 16 changed real things (e.g. `proxy.ts` replacing `middleware.ts`). |

If any instruction below seems to contradict one of these four files,
the file wins — update this CLAUDE.md instead of silently overriding it.

## Pillar 1 — Stack & non-negotiables

- **Next.js 16, App Router, `output: "export"`** — fully static site, no
  Node server at runtime. Every page must be statically generatable. This
  has real consequences threaded through every pillar below (especially
  Pillar 4 and 7) — don't reach for `getServerSideProps`-style patterns,
  Route Handlers that need a live server, or `next/image`'s server
  optimizer. None of them work in static export.
- **TypeScript throughout.** No `.js`/`.jsx` files. No `any` — if a shape
  is genuinely unknown, write the actual union/interface.
- **Tailwind CSS v4, CSS-first config.** Theme tokens live in
  `src/app/globals.css` under `@theme inline { ... }`, not in a
  `tailwind.config.js` file with a `theme.extend` object — v4 deprecated
  that pattern. If you find yourself reaching for a `tailwind.config.ts`
  to add a color or radius, stop — add the CSS variable to `globals.css`
  instead.
- **shadcn/ui conventions, manually maintained.** The shadcn CLI's
  `init`/`add` commands may hang or error in this sandboxed environment
  (confirmed: `shadcn@latest init` both rejected an unsupported `-b` flag
  value and then hung past the command timeout on a retry). Don't burn
  time fighting the CLI — components in `src/components/ui/` are written
  by hand following shadcn's exact patterns (CVA variants, `data-slot`
  attributes, `cn()` merging). If the CLI works in your environment, feel
  free to use it for *new* components — just match the existing file
  style for consistency.
- **lucide-react**, `strokeWidth={1.75}` everywhere (never the default
  `2` — see design.md §10). Sizes: 14–16 inline, 18–20 section headers,
  28–32 empty-state/hero icons.
- **JSON as the database.** No SQL, no ORM, no API routes for data. See
  Pillar 3 for the full data pipeline.
- **Netlify, static deploy.** `netlify.toml` is already configured
  (`npm run build` → publish `out/`). Don't add Netlify Functions, Edge
  Functions, or anything implying a server unless the project's scope
  explicitly changes.
- **RTL Arabic only.** This is not an i18n project with a language
  switcher — it ships exactly one language and one direction. `dir="rtl"`
  and `lang="ar"` are set once, on `<html>`, in the root layout. Don't
  build a locale system for a site that only ever needs one locale.
- **Mobile-first.** Every component is designed for a ~375–430px
  viewport first, then enhanced upward with `md:`/`lg:` prefixes — never
  the reverse. The reference mockup is phone-shaped for a reason.

## Pillar 2 — Design system contract

Full detail lives in `design.md`; this is the enforcement summary.

1. **One accent color, always via CSS variable.** Never hardcode a hex
   that happens to match today's accent. Always
   `bg-[var(--accent-color)]`, `text-[var(--accent-color)]`, etc. The
   accent is user-configurable at runtime (`src/lib/accent-color.ts`,
   `AccentColorProvider`) — a hardcoded hex silently desyncs the moment
   someone picks a different swatch. **Before any PR/commit, grep for
   stray hex literals** that match the current default
   (`grep -rn "#E8503A" src/` excluding `lib/accent-color.ts` and
   `globals.css`) and replace any hit with the variable.
2. **Two-tier button rule.** The primary "commit this data" action
   (Save/Add/Submit) is near-black (`variant="primary"` in
   `components/ui/button.tsx`), never accent-colored. The accent is
   reserved for recurring small CTAs, nav active-state, and tab
   selection (`variant="accent"`). Audit every new submit button against
   this rule — it is the single most common visual inconsistency in
   projects built off this design system.
3. **Glass, not borders.** Card-level surfaces use `.glass-card` /
   `.glass-card-dark` (defined in `globals.css`), which use **inset
   `box-shadow`, not the `border` shorthand** — this is deliberate so a
   consumer can still add `border-l-4 border-l-[var(--accent-color)]`
   for an accent-bar treatment without the glass class's border
   silently winning the cascade fight. Never add `border: ...` directly
   to `.glass-card`.
4. **Radius by tier**, not by feel: `rounded-xl` inputs/buttons,
   `rounded-2xl` nested boxes/pills, `rounded-3xl` cards. All derived
   from one `--radius` variable in `globals.css` — don't hand-pick
   arbitrary `rounded-[Npx]` values.
5. **Background blobs for glass depth.** Glass over a flat background
   looks flat. `<BackgroundBlobs />` (in `src/components/layout/`) is
   already wired into every page — don't remove it, and don't add a
   second competing background treatment.
6. **Motion** (if/when `framer-motion` is added): 150–250ms for
   micro-interactions, 400–600ms for entrances, `easeOut` almost
   everywhere. Nothing UI-chrome-related should run longer than 600ms.
7. **Empty states trigger off existence, not off a falsy value.** E.g.
   "no meals match this filter" should check `filtered.length === 0`,
   which is correct here — but if a future feature has a field that can
   legitimately be zero (a count, a price), don't reuse the same
   `!value` check for its empty state or a real zero gets treated as
   "not filled in yet."

## Pillar 3 — Data layer: from raw JSON to typed content

**The chain is: raw scraped JSON → one-time transform script → clean
JSON → typed accessor functions → components.** Never skip a link in
this chain.

1. **`src/data/iraqi_meals.raw.json`** — the original scrape. Arabic keys
   with spaces (`"meal name"`), one long free-text paragraph for the
   method, an external `imgUrl` pointing at `cpcdn.com`. **Never import
   this file directly into a component.**
2. **`scripts/transform-meals.mjs`** — run via `npm run data:transform`.
   Converts the raw shape into `src/data/meals.json`:
   - Generates a stable `slug` (`meal-01` … `meal-27`) per entry, used for
     `/meals/[slug]` routing. Index-based, not transliterated from
     Arabic — transliteration is lossy and the raw order never changes,
     so this is simpler and just as stable.
   - Rewrites the image reference to a **local path**
     `/images/meals/<slug>.jpg`. This script does **not** download
     images — per project decision, real photos are supplied manually
     into `/public/images/meals/` (see `public/images/meals/README.md`
     for the exact naming convention). The transform only standardizes
     the *path string* so no component ever references an external URL.
   - Splits the single `methodofcook` paragraph into a `steps[]` array on
     sentence-ending punctuation, since the detail page (mockup screen 3)
     needs numbered steps, not a wall of text.
   - Normalizes field names to clean camelCase (`timeOfCook`, `portion`,
     `ingredients`, `category`).
   - **Re-run this script any time the raw JSON changes** (meals
     added/removed/edited) — `meals.json` is a build artifact, not a
     hand-edited file. If you edit `meals.json` directly, the next
     transform run will silently overwrite your edit.
3. **`src/types/meal.ts`** — the canonical `Meal` interface. Import this
   type everywhere instead of letting TypeScript infer shape from the
   JSON import.
4. **`src/lib/meals.ts`** — the only place components should get data
   from: `getAllMeals()`, `getMealBySlug()`, `getAllSlugs()`,
   `getAllCategories()`, `getMealsByCategory()`, `searchMeals()`. This
   indirection means if the data source ever moves (a CMS, an API), only
   this one file changes — not every component that reads meal data.

### On categories — a deliberate decision, don't "fix" it later

The real dataset has **10 distinct category strings**
(أكلة باللحم, أكلة بالدجاج, أكلة بالسمك, أكلة خضار, شوربة / تشريب,
مقبلات وسلطات, أكلة جانبية / نشويات, حلويات, حلويات / مخبوزات, مخبوزات) —
including near-duplicates like `حلويات` vs `حلويات / مخبوزات` vs
`مخبوزات`. The design mockup shows only 4 filter tabs. **This is
intentional, by project decision: the mockup is a layout/vibe reference,
not a literal data spec.** `getAllCategories()` derives the filter list
live from whatever's actually in `meals.json`, in first-seen order. Do
**not**:
- Hardcode a 4-item category list anywhere "to match the mockup."
- Silently merge the near-duplicate sweets categories without being
  asked — that's a content decision (are `حلويات` and `مخبوزات` actually
  the same thing to a home cook, or meaningfully different?), not a
  technical one, and it's the project owner's call.

If category cleanup is ever requested, do it **in the raw JSON or the
transform script**, never by hardcoding a mapping deep inside a UI
component — that guarantees the mapping rots the next time a meal is
added.

### On images — local-only, graceful degradation

`next.config.ts` sets `images.unoptimized = true`. This is **mandatory**,
not a preference: `output: "export"` (static export) has no server to run
Next's image optimizer, and `next build` fails outright in export mode
without this flag. Pre-compress source photos yourself before adding them
— Next will not optimize them for you in this mode.

`MealImage` (`src/components/meals/meal-image.tsx`) wraps `next/image`
with an `onError` fallback to a calm placeholder (icon + "صورة" label) —
**never a broken-image icon.** This means the site is safe to build and
deploy at any point before every one of the 27 photos exists locally; the
grid degrades gracefully instead of looking broken. Don't remove this
fallback "because the images will all be there eventually" — they won't
always be, and a build-time `next/image` error on a missing local file is
much worse UX than a placeholder.

## Pillar 4 — Static export realities (read this before any dynamic route)

`next.config.ts` has `output: "export"`. This single setting changes a
lot of normal Next.js advice:

- **Every dynamic route (`[slug]`) needs `generateStaticParams`.**
  `src/app/meals/[slug]/page.tsx` already has this, returning every slug
  from `getAllSlugs()`. If you add a new dynamic route, you must add this
  export too, or `next build` fails. There is no server-side fallback —
  a slug not enumerated here 404s in production even if it technically
  exists in `meals.json`.
- **No Server Actions, no Route Handlers that mutate state, no
  middleware that needs to run per-request.** Static export has no
  server process listening after build. If a future feature needs
  "submit a form to a backend," that's an external service (Netlify
  Forms, a third-party API called client-side) — not an API route in
  this repo.
- **`useSearchParams()` requires a `<Suspense>` boundary**, or `next
  build` throws at build time. `src/app/meals/page.tsx` already wraps
  `<MealsBrowser>` in `Suspense` for this exact reason — copy this
  pattern for any other client component reading search params.
- **`next/image`'s built-in optimizer is unavailable** (see Pillar 3)
  — `unoptimized: true` is required, not optional, in `next.config.ts`.
- **Test with the real static output, not just `next dev`.** `next dev`
  runs a dev server that papers over some static-export-only failures.
  Before trusting a change, run `npm run build` and serve `/out` directly
  (e.g. `npx serve out` or `python3 -m http.server --directory out`) to
  catch issues that only show up in the actual exported HTML.

## Pillar 5 — RTL & Arabic-specific correctness

- `<html lang="ar" dir="rtl">` is set **once**, in `src/app/layout.tsx`.
  Never set `dir` per-component or per-page — RTL is a document-level
  property, fighting it locally causes mixed-direction bugs that are
  miserable to debug.
- Tailwind's logical properties (`ms-`/`me-` for margin-start/end,
  `ps-`/`pe-` for padding) automatically flip for RTL — prefer them over
  physical `ml-`/`mr-` when a value is direction-dependent (e.g. an icon
  sitting "before" text in reading order). Physical values are fine when
  something is genuinely direction-agnostic (e.g. vertical spacing).
- **Numerals and prices often read better forced LTR** even inside RTL
  flow — a phone number or a price like "12.99 ر.س" can look garbled if
  it's allowed to participate in bidi reordering. The `.ltr-numerals`
  utility class in `globals.css` (`direction: ltr; unicode-bidi: isolate`)
  exists for exactly this — already applied to cook-time and portion
  values in the meal card and detail page. Use it for any other
  standalone numeral/measurement string.
- Arabic sentence-final punctuation (`.`, `؟`, `!`) is the same set of
  characters as Latin scripts — the `transform-meals.mjs` step-splitter
  relies on this and needs no special Arabic NLP. Don't over-engineer
  Arabic text processing where the punctuation behaves identically to
  English.
- This dataset's Arabic has **no diacritics (tashkeel)** — `searchMeals()`
  uses a naive `.includes()` substring match, which is correct for this
  data. If a future data source introduces diacritics, naive matching
  will start missing results (a user typing without tashkeel won't match
  stored text with tashkeel) — revisit with Unicode normalization
  (`NFKD` + strip combining marks) at that point, not before.
- **Test the actual layout in RTL, not just the text direction.** Icons
  that imply directionality (arrows, chevrons) need to visually point the
  *correct* way for RTL reading order — e.g. a "back" arrow in this app
  correctly uses `ArrowRight` (pointing toward where "back" is, on the
  right, in RTL), not `ArrowLeft`. Don't copy-paste LTR icon choices
  without checking they still make sense mirrored.

## Pillar 6 — Component patterns already established

Follow these patterns for any new component rather than introducing a
different convention:

- **Data fetching:** Server Components (the default in App Router) call
  `src/lib/meals.ts` functions directly and pass data down as props.
  Client Components (`"use client"`) that need interactivity (search,
  filter state) receive already-fetched data as props rather than
  re-fetching — see `MealsBrowser` receiving `meals`/`categories` from
  its parent Server Component.
- **CVA for variant components.** `Button` and `Badge` use
  `class-variance-authority` for variant/size props, matching shadcn's
  own convention — don't switch to a different variant pattern (e.g.
  plain ternaries) for new primitives; it fragments the API surface.
- **`cn()` for all conditional classes** (`src/lib/utils.ts`, wraps
  `clsx` + `tailwind-merge`) — never string-concatenate classNames by
  hand, and never use plain `clsx` without the `tailwind-merge` pass, or
  conflicting utility classes (e.g. two different `bg-*` values) won't
  resolve correctly.
- **Image components always have a graceful failure path** — see
  `MealImage`. Any future image-bearing component (e.g. a user avatar,
  if accounts are ever added) should follow the same onError-to-placeholder
  pattern, never a raw `next/image` with no fallback.

## Pillar 7 — Performance checklist

- **Static export is already the biggest performance win** — every page
  is pre-rendered HTML, no server round-trip, ideal for Netlify's CDN.
  Don't undermine this by reintroducing client-side data fetching for
  content that's already known at build time (i.e. don't `fetch()` the
  meals JSON client-side — it's already bundled).
- **`priority` on `next/image`/`MealImage` only for above-the-fold
  images** — the hero image on the home page and the first couple of
  featured cards, per the existing pattern in `page.tsx`. Marking every
  image `priority` defeats the purpose (it disables lazy loading
  entirely) and will hurt LCP, not help it.
- **`sizes` prop on every `MealImage` in a grid** — already set
  (`"(max-width: 640px) 50vw, 25vw"` for 2-column mobile grids) so the
  browser doesn't request a larger image than it'll actually display.
- **Pre-compress source photos** before dropping them into
  `/public/images/meals/` — see Pillar 3. With image optimization
  disabled (a static-export requirement), whatever ships in `/public` is
  exactly what downloads in production, byte for byte.
- **Keep the bundle's only heavy dependency, `lucide-react`, tree-shaken**
  — always `import { IconName } from "lucide-react"` (named imports),
  never `import * as Icons from "lucide-react"`, or the whole icon set
  ends up in the client bundle.
- **Avoid unnecessary `"use client"` boundaries.** Only `meals-browser.tsx`,
  `meal-image.tsx`, `bottom-nav.tsx`, and the accent-color provider need
  client-side interactivity. Every page (`page.tsx`) and `MealCard` are
  Server Components by default — keep it that way. Adding `"use client"`
  to a component that doesn't need state/effects/event handlers only
  grows the client JS bundle for no benefit.

## Pillar 8 — Debugging guide (problems you will likely hit)

| Symptom | Cause | Fix |
|---|---|---|
| `next build` fails with an error about `generateStaticParams` or dynamic routes | A `[param]` route is missing `generateStaticParams`, required by `output: "export"` | Add the export, returning every valid param value — see `meals/[slug]/page.tsx` for the pattern |
| Build error mentioning `useSearchParams()` needing a Suspense boundary | A Client Component reads search params outside `<Suspense>` | Wrap the component in `<Suspense fallback={...}>` at the call site, as done in `meals/page.tsx` |
| `next/image` throws about missing `loader`/optimization config in export mode | `images.unoptimized` not set, or was removed from `next.config.ts` | Restore `images: { unoptimized: true }` — it's required, not optional, under `output: "export"` |
| A meal's image shows the placeholder icon instead of the photo | The expected file (`/public/images/meals/<slug>.jpg`) doesn't exist yet, or the slug in `meals.json` doesn't match the filename | Confirm the slug via `src/data/meals.json`, add the file with that exact name — this is expected behavior pre-launch, not a bug, per `MealImage`'s designed fallback |
| Accent color flashes the wrong color for a split second on page load | The pre-hydration `<script>` in `layout.tsx`'s `<head>` was removed, or `AccentColorProvider` was moved to render before it | The inline script **must** run before first paint and before the provider mounts — don't move accent-color initialization into a `useEffect` |
| A filter/search change doesn't update the visible grid | State is being read from `meals.json` directly in a component instead of through `src/lib/meals.ts`, or filtering logic was duplicated in two places computing slightly different results | Centralize all meal-list derivation (filtering, search) through one path — don't compute "filtered meals" two different ways in two components on the same screen (see design.md §11.5 — this is a recurring class of bug) |
| Hardcoded hex color spotted where the accent should be | A new component used a literal hex instead of `var(--accent-color)` | `grep -rn "#E8503A" src/` (swap in whatever the current default is) excluding `lib/accent-color.ts` and `globals.css` — replace any hit |
| `npm run data:transform` overwrote a manual edit to `meals.json` | `meals.json` is a generated build artifact, not a source file | Make the edit in `iraqi_meals.raw.json` (or the transform script's logic) instead, then re-run the transform |
| RTL layout looks mirrored incorrectly (icons pointing the wrong way, text overflowing oddly) | A component used physical `ml-`/`mr-`/`left-`/`right-` where a logical or RTL-aware value was needed, or an icon's inherent directionality wasn't reconsidered for RTL | Swap to logical properties (`ms-`/`me-`) where the value should flip with direction; manually re-check any directional icon (arrows, chevrons) for whether it still reads correctly in RTL |
| shadcn CLI (`npx shadcn@latest add ...`) hangs or errors | Confirmed flaky/hanging in this sandboxed dev environment | Don't retry indefinitely — write the component by hand following the existing `button.tsx`/`badge.tsx` style (CVA variants, `data-slot`, `cn()`), or try the CLI in a different (non-sandboxed) terminal if available |

## Pillar 9 — What NOT to do

- Don't add a CMS, database, or backend API for content. The brief is
  static-from-JSON; if real dynamic data (user accounts, comments,
  ratings) becomes a requirement, that's a scope change worth flagging
  explicitly, not something to quietly half-implement with
  `output: "export"` still set.
- Don't add a language switcher or i18n library (`next-intl`, etc.) for
  a single-language Arabic site — that's solving a problem this project
  doesn't have, at the cost of real complexity.
- Don't "fix" the 10-vs-4 category mismatch by hardcoding a 4-category
  filter to match the mockup. See Pillar 3.
- Don't hand-edit `src/data/meals.json`. It's regenerated from the raw
  JSON. See Pillar 3.
- Don't introduce `getServerSideProps`/the Pages Router, or any
  App Router pattern that assumes a persistent server process. This is
  static export, permanently, unless the project's hosting requirements
  change.
- Don't mark every image `priority`, add `border` to `.glass-card`, or
  introduce a second hardcoded accent-like color "just for this one
  card" — these are the exact anti-patterns design.md calls out as
  recurring bugs in projects using this design system.

## Pillar 10 — Quick command reference

```bash
npm run dev            # local dev server (Turbopack)
npm run build          # production build -> static export in /out
npm run lint           # eslint
npm run data:transform # regenerate src/data/meals.json from the raw JSON

# Smoke-test the actual static export (closer to production than `next dev`):
npx serve out
# or
python3 -m http.server 8080 --directory out
```

Deploy: push to the connected Git repo; Netlify builds via `netlify.toml`
(`npm run build`, publishes `out/`). No environment variables or secrets
are required for this project's current scope — if any are added later
(an API key, etc.), document them here and in a `.env.example`, never
commit the actual `.env`.
